# Pars_GAZ — парсинг docx-документов для нейронки (RAG)

Конвертирует `.docx` (в первую очередь — документы КонсультантПлюс) в JSON-фрагменты
со структурой, оптимальной под embedding + LLM.

## Установка

```bash
pip install -r requirements.txt
```

## Запуск

```bash
python parse_docs.py                 # парсит всё из ./DOKI, кладёт в ./OUT
python parse_docs.py DOKI OUT        # явно
python parse_docs.py DOKI/file.docx OUT
```

Для каждого `<имя>.docx` создаётся `OUT/<имя>.json`. Если в документе есть
картинки/блок-схемы — рядом с json'ом сохраняются файлы вида
`schema{N}_{имя_документа}_{fragment_index}.png`.

## Структура фрагмента

```json
{
  "fragment_index": 12,
  "type": "text",
  "text": "Утвердить прилагаемый профессиональный стандарт...",
  "header": "I. Общие сведения > Группа занятий:",
  "page_number": 1,
  "point": "1.",
  "note": "<1> Общероссийский классификатор занятий.",
  "url": "consultantplus://offline/ref=...",
  "fz_number": "от 21.07.1997 N 116-ФЗ",
  "fz_text": "О промышленной безопасности опасных производственных объектов",
  "images": ["schema1_11. Приказ Минтруда…_12.png"],
  "is_indexable": true
}
```

Поля:

| поле | смысл |
|---|---|
| `fragment_index` | порядковый номер фрагмента в документе |
| `type` | `text` / `table` / `footnote` / `title` — см. ниже |
| `text` | для `table` — markdown-таблица (≤ **3000** символов); для остального — текст (≤ **512** символов) |
| `header` | путь по иерархии заголовков, склеенный через ` > ` |
| `page_number` | страница (`lastRenderedPageBreak` + явные page-break + эвристика по объёму текста) |
| `point` | маркер пункта в начале параграфа: `1.`, `2.1.`, `1)`, `а)`, `II.` |
| `note` | примечание / сноска `<N>`, привязанная к фрагменту |
| `url` | гиперссылки (`http://`, `https://`, `consultantplus://`), через `\n` если несколько |
| `fz_number` | номер ФЗ из текста, например `от 21.07.1997 N 116-ФЗ` |
| `fz_text` | название ФЗ (то, что в кавычках сразу после номера) |
| `images` | имена PNG-файлов, лежащих рядом с json'ом |
| `is_indexable` | `false` — служебный/юридический блок, **не индексировать** в RAG (см. ниже) |

### Что отсеивается (`is_indexable=false`)

Юридические/служебные блоки засоряют top-k retrieval. Парсер их **оставляет
в JSON** (чтобы можно было показать как контекст или для отладки), но
помечает `is_indexable=false`. Индексер должен брать только `is_indexable=true`.

В индекс **НЕ** идут:

- `type="title"` — преамбула документа (министерство / № приказа / дата / название);
- «Зарегистрировано в Минюсте России …», «Зарегистрировано Министерством юстиции …»;
- «Список изменяющих документов (в ред. Приказов …)»;
- «В соответствии с … приказываю:» (вступительная формула приказа);
- «Утвердить Правила / Порядок / Положение / прилагаемый …»;
- «Признать утратившими силу:» **и весь следующий перечень** «приказ Министерства … (зарегистрирован Министерством юстиции … N XXXXX);»;
- «Утратил силу. — Приказ Минтруда России от … N …н.»;
- подписи: «Министр», ФИО формата `А.О.КОТЯКОВ`;
- шапки приложения: «Приложение к приказу …», «Российской Федерации», «от 14 июля 2021 г. N 467н»;
- редакционные пометки целиком: «(в ред. Приказа Минтруда России от 27.02.2023 N 101н)».

Пример отсева:

```
файл 11 (профстандарт): 70 фрагментов → в индекс 64, отсеяно  6
файл 12 (правила фин.обеспечения): 149 фрагментов → в индекс 130, отсеяно 19
```

## Типы фрагментов

| `type` | что это | лимит длины |
|---|---|---|
| `title` | шапка документа (министерство, № приказа, дата, название) — один блок на документ | без лимита |
| `text` | обычный параграф | **≤ 512** символов |
| `table` | таблица в markdown-формате (с шапкой `\| ... \|` и разделителем `\|---\|`) | **БЕЗ лимита** — 1 таблица = 1 фрагмент |
| `footnote` | сноска `<N>` внизу документа | ≤ 512 символов |

Логика chunking: **по logical block, а не по символам**. Таблица — атомарная
единица retrieval (никогда не режется). Параграф — атомарная единица.
Только если параграф ОЧЕНЬ длинный, он режется по границам предложений
с soft-границей в 480 символов.

### Cross-section dedup для таблиц

Таблицы дедуплицируются **глобально по содержимому**, без учёта `header`.
Это лечит типичный кейс пустого шаблона-формы в КонсультантПлюс (например,
«Происхождение трудовой функции» с пустыми ячейками для отметок),
который иначе клонировался бы в каждый раздел `3.1.1`, `3.1.2`, ..., `3.4.4`.

### Очистка таблиц (нормализация)

Перед рендером в markdown матрица таблицы чистится:

1. Строки целиком из ячеек-пояснений в скобках (`(наименование)`, `(код)`,
   `(код ОКЗ <1>)`) и из `—`/`-`/пустых — выкидываются.
2. Колонки, полностью состоящие из таких ячеек — удаляются.
3. Отдельные декоративные ячейки заменяются на пустую строку.
4. Подряд идущие дубли строк (артефакт `vMerge`) и одинаковые соседние
   ячейки в строке (артефакт `gridSpan`) — схлопываются.
5. Если в итоге осталась только одна строка-шапка без данных — таблица
   полностью выкидывается.

## Таблицы (clean-pipeline)

1. **Парсинг матрицы** с учётом `<w:gridSpan>` (colspan) и `<w:vMerge>` (rowspan).
   Объединённые ячейки разворачиваются корректно, vMerge-continue наследует
   текст у restart-ячейки сверху.
2. **Классификация таблицы**:
   - `single` — одна осмысленная колонка → plain text;
   - `kv`     — 2 эффективные колонки (label | value) **или** 3+ колонок, где
                первая колонка везде короткая (≤ 80 символов) и играет роль
                лейбла → markdown с шапкой `Параметр | Значение | ...`;
   - `data`   — 3+ колонок с короткой header-строкой → markdown с этой
                шапкой;
   - `mixed`  — fallback: первая строка как header, остальные как data.
3. **Одна таблица = один markdown-блок** (`type="table"`). Лимит 3000
   символов; огромные таблицы режутся ПО СТРОКАМ с дублированием шапки
   в каждой части — никогда посредине строки.
4. **Дубли соседних строк** (последствие vMerge-разворота) сворачиваются.
5. **Дубли ячеек внутри строки** (последствие gridSpan-разворота) сворачиваются.
6. Символ `|` внутри ячеек экранируется как `\|`, переносы строк — заменяются
   на пробел (чтобы markdown-таблица не сломалась).

## Semantic chunking и дедуп

- **Semantic merge**: соседние блоки из ОДНОГО `header` (без `point` и без
  картинок) склеиваются в один, пока суммарная длина ≤ 512. Меньше мелких
  бесполезных фрагментов, лучше retrieval.
- **Глобальный dedup**: блоки с одинаковым `header` и одинаковым
  (нормализованным) `text` схлопываются в один; картинки из дубликата
  переносятся в оставшийся блок. Текстуальные совпадения **в разных
  разделах** не считаются дублями — это разные контексты.
- **Garbage cleanup**: мелкие фрагменты (`type=text`, длина < 30 символов,
  без `point`/`url`/`note`/`images`) объединяются с соседними того же
  `header`, либо удаляются.

## Картинки

Любая `<a:blip>` внутри `<w:drawing>` сохраняется как PNG в выходную папку
с именем `schema{N}_{json_name}_{fragment_index}.{ext}` и попадает в поле
`images` ближайшего фрагмента. Имя fragment_index'а в файле = индекс
первого чанка блока, к которому привязано изображение.

Поддерживаемые форматы — png/jpg/jpeg/gif/bmp/tiff. EMF/WMF сохраняются
с расширением `.png` как заглушка (для них нужна доп. конвертация).

## Что НЕ парсится сейчас

- EMF/WMF векторные изображения (сохраняются «как есть» под `.png` — могут не открыться).
- Сноски Word через `<w:footnoteReference>` (в КонсультантПлюс этого нет,
  они делают inline-сноски `<N>` обычным текстом — это парсится).
- Многоколоночная вёрстка одной страницы не учитывается отдельно.

## Тонкая настройка

В `parse_docs.py` сверху:

```python
MAX_TEXT_CHARS = 512      # лимит длины text для обычных параграфов
SOFT_CHARS = 480          # граница, после которой ищем конец предложения
CHARS_PER_PAGE = 2800     # эвристика страниц, если в docx нет page-break'ов
GARBAGE_MIN_CHARS = 30    # text короче этого без метаданных = мусор
```

**Важно**: для таблиц лимит **НЕ задаётся** — одна таблица всегда = один
фрагмент, независимо от размера (для retrieval это критично).
